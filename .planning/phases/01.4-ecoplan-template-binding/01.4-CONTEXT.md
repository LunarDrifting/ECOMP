# Phase 1.4: ECOPlan Template Binding - Context

**Gathered:** 2026-02-27
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Add Prisma schema support for `ECOPlan` as the domain model that binds an `ECO` instance to a `TemplateVersion`. This phase is schema and migration work only.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Schema-only changes; no business logic.
- `ECOPlan` must:
  - Reference `ECO` as a 1:1 relationship
  - Reference `TemplateVersion`
  - Include `tenantId`
  - Include `createdAt` and `updatedAt`
  - Enforce 1:1 with `@@unique([ecoId])`
  - Include `@@index([tenantId])`
  - Include `@@index([templateVersionId])`
- `ECO` must include inverse relation to `ECOPlan`.
- Do not modify existing model fields except required inverse relations.
- Do not rename existing fields.
- `TemplateVersion` -> `ECOPlan` relation must not use cascade deletion (use `Restrict` or Prisma default behavior).
- Sequential execution only.

### Claude's Discretion
- Index and unique-constraint details needed to enforce 1:1 ECO binding and query efficiency.
- Relation naming details if explicit relation names are needed for Prisma clarity.

</decisions>

<specifics>
## Specific Ideas

- Enforce ECO 1:1 via unique constraint on `ecoId` in `ECOPlan`.
- Keep tenant path explicit with `tenantId -> Tenant.id`.
- Keep TemplateVersion binding auditable and stable through foreign key relation.
- Add explicit indexes for `tenantId` and `templateVersionId` on ECOPlan.

</specifics>
