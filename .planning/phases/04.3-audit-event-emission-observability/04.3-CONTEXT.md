# Phase 4.3: AuditEvent Emission (Deterministic Observability, No Behavior Drift) - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Emit immutable audit events for key write-side actions to improve observability while preserving existing business behavior and outcomes.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes (must use existing `AuditEvent` model as-is).
- Must remain deterministic.
- Must not change endpoint success/failure semantics.
- Must not leak secrets or PII.
- Must enforce tenant isolation.
- Must use `tenantDb` wrapper only.
- Must not introduce background workers.
- Must keep transaction behavior correct; tx-bound operations emit within same tx where appropriate.
- Execution remains sequential.

### Existing Model Constraint
- `AuditEvent` currently requires `ecoId` and does not have a `tenantId` column.
- Tenant scoping for audit emission must be enforced by deriving/validating `ecoId` via tenant-scoped lookup before insert.
- No cross-tenant event creation allowed.

### Required Audit Coverage
1. Instantiation:
   - `INSTANTIATE_ATTEMPT`
   - `INSTANTIATE_SUCCESS` or `INSTANTIATE_REJECTED`
   - payload: `ecoId`, `templateVersionId`, counts, status
   - rejected payload uses reason code only (`BLUEPRINT_INVALID`, `MISSING_DEFINITIONS`, etc.)
2. Task completion:
   - `TASK_COMPLETE_ATTEMPT`
   - `TASK_COMPLETE_SUCCESS` or `TASK_COMPLETE_REJECTED` or `TASK_COMPLETE_NOOP`
   - payload: `taskId`, `actorId`, result status, `tasksUnblocked`
3. Dependency cascade summary:
   - `CASCADE_RESOLVE`
   - payload: `startingTaskId`, `tasksUnblocked`, optional `unblockedTaskIds`, status
4. Approval creation:
   - `APPROVAL_CREATE_ATTEMPT`
   - `APPROVAL_CREATE_SUCCESS` or `APPROVAL_CREATE_REJECTED`
   - payload: `taskId`, `actorId`, `decision`
5. Gate enforcement failure:
   - `GATE_PRECONDITION_FAILED`
   - payload: `taskId`, optional gate ids, minimal reason only (no raw condition JSON)

### Payload Safety Rules
- JSON payload limited to IDs, enums, counts, and status/reason strings.
- Exclude names/emails/request bodies/raw gate JSON/connection strings/secrets.
- Keep payload under 4KB.

### Transaction/Failure Rules
- For tx-scoped operations (`markTaskDone` + cascade), emit audit records using same tx client.
- Read-only projection emits no audit events.
- Emission is best effort and must not alter primary behavior:
  - on insert failure, catch and continue deterministically
  - inside tx, do not abort primary mutation due to audit insert failure

### Claude's Discretion
- Exact event payload key names within safety constraints.
- Exact reason code taxonomy for rejections, as long as deterministic and bounded.

</decisions>

<specifics>
## Specific Ideas

- Add helper `emitAuditEvent({ tenantId, ecoId?, taskId?, actorId?, eventType, payload, tx? })`.
- Resolve `ecoId` from `taskId` using tenant-scoped reads when not provided.
- Centralize payload sanitization/size enforcement in helper.
- Keep event emission calls close to success/rejection branch points in services.

</specifics>

<deferred>
## Deferred Ideas

- Streaming/export of audit events.
- Structured analytics pipeline.
- Rich reason metadata beyond safe enums/codes.

</deferred>
