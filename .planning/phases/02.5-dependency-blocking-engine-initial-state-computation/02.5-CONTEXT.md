# Phase 2.5: Dependency Blocking Engine (Initial State Computation) - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Compute initial `Task.state` values for newly instantiated runtime tasks based on incoming runtime dependencies.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce tenant isolation strictly.
- Must use `tenantDb` wrapper only (no raw Prisma in service/route).
- Must not implement dynamic state transitions yet.
- Must not evaluate gates or approvals.
- Must not mutate existing tasks (idempotent branch untouched).
- Sequential execution only.

### Behavior Requirements
1. After runtime dependencies are created, compute for newly created tasks:
   - If task has zero incoming dependencies -> `state = NOT_STARTED`
   - If task has one or more incoming dependencies -> `state = BLOCKED`
   - Incoming dependency presence must be determined from runtime `Dependency` rows (`fromTaskId`/`toTaskId`) only.
   - Use runtime dependency rows created in this instantiation, not blueprint definition edges.
2. Apply this logic only during first instantiation (not idempotent no-op path).
3. Do not re-evaluate state for pre-existing tasks.
4. Update task state deterministically after dependency creation.
5. Return structured result including:
   - `blockedTasks` count
   - `readyTasks` count

### Claude's Discretion
- Efficient computation approach for incoming edge counts (e.g., in-memory from created dependency set vs tenantDb query), as long as deterministic and tenant-scoped.
- Route response wording while preserving thin route/service delegation.

</decisions>

<specifics>
## Specific Ideas

- Add tenantDb helper for bulk task state updates constrained to tenant + task ids.
- Use runtime `Dependency` rows created during this instantiation to derive incoming dependency counts per created task.
- Keep idempotent no-op branch returning zero blocked/ready counts.

</specifics>

<deferred>
## Deferred Ideas

- Runtime state transition engine beyond initial instantiation.
- Gate/approval-aware blocking rules.
- Recalculation for existing tasks after dependency graph mutations.
- Any schema/model migrations.

</deferred>
