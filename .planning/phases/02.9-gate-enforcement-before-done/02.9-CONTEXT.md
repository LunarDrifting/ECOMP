# Phase 2.9: Gate Enforcement Before DONE - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Enforce `PRECONDITION` gates in `markTaskDone` before allowing task DONE transitions.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce tenant isolation strictly.
- Must use `tenantDb` wrapper only (no raw Prisma in route).
- Must not introduce background workers.
- Must preserve idempotency.
- Must not implement `POSTCONDITION` behavior yet.
- Sequential execution only.

### Behavior Requirements
1. In `markTaskDone`:
   - After approval enforcement but before DONE mutation:
     - Load tenant-scoped `Gate` rows for task where `type === PRECONDITION`.
     - Evaluate condition JSON using minimal evaluator:
       - If `condition.allow === true` -> pass
       - Otherwise -> fail.
2. If any PRECONDITION gate fails:
   - Reject with deterministic `409`.
   - Do not mutate state.
   - Do not call resolver.
3. If all pass:
   - Continue DONE update.
   - Call `resolveDependenciesForTask`.
4. Preserve:
   - BLOCKED rejection.
   - Approval enforcement.
   - Already-DONE idempotency.
5. Return structured result including:
   - `taskMarkedDone`
   - `tasksUnblocked`
   - `unblockedTaskIds`
   - `approvalCheckPassed`
   - `gateCheckPassed`

### Claude's Discretion
- Deterministic error text for gate failure as long as route maps it to 409.
- Whether zero PRECONDITION gates yields `gateCheckPassed=true` (recommended) or a separate noop signal, as long as deterministic.

</decisions>

<specifics>
## Specific Ideas

- Add tenantDb helper to list task gates filtered to `PRECONDITION`.
- Keep gate evaluator intentionally minimal (`condition.allow === true`) and explicit in code comments.
- Maintain existing control flow ordering: blocked check -> already-done idempotency -> approval check -> gate check -> done mutation -> resolver.

</specifics>

<deferred>
## Deferred Ideas

- POSTCONDITION evaluation behavior.
- Rich expression evaluators for gate conditions.
- Async/background orchestration for gate resolution.
- Any schema/model migrations.

</deferred>
