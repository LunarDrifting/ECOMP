# Phase 2.7: Explicit DONE Transition Endpoint - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Introduce a canonical service method and API endpoint to mark a task as `DONE` and trigger runtime dependency resolution.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce tenant isolation strictly.
- Must use `tenantDb` wrapper only (no raw Prisma in route).
- Must not evaluate approvals or gates yet.
- Must not introduce background workers.
- Must preserve idempotency.
- Sequential execution only.

### Behavior Requirements
1. Add service method `markTaskDone({ tenantId, taskId })`.
2. Validate:
   - Task exists for tenant.
   - Task is not already `DONE`.
3. Update `Task.state` to `DONE`.
4. Call `resolveDependenciesForTask({ tenantId, taskId })`.
5. Return structured result:
   - `taskMarkedDone` (boolean)
   - `tasksUnblocked`
   - `unblockedTaskIds`
6. Add endpoint: `POST /api/tasks/{id}/complete`.
7. Route must remain thin and Prisma-free.

### Claude's Discretion
- Exact idempotent response semantics when task is already `DONE` (e.g., conflict vs no-op), as long as deterministic.
- HTTP status mapping for validation/transition errors.

</decisions>

<specifics>
## Specific Ideas

- Keep completion logic in service layer and call existing standalone `resolveDependenciesForTask` service method.
- Add tenantDb helper for single task state update to `DONE` under tenant scope.
- Return resolver payload directly from completion service for deterministic downstream effect visibility.

</specifics>

<deferred>
## Deferred Ideas

- Approval/gate validations before allowing DONE.
- Async transition orchestration or event bus hooks.
- Additional transition endpoints for other states.
- Any schema/model migrations.

</deferred>
