# Phase 4.1: Workflow Projection Enrichment (Read-Only, Batched, Deterministic) - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Enrich the existing workflow projection so clients can render full workflow graph state directly from read model output, without client-side recomputation of engine logic.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Read-only only (no writes, no mutations).
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce tenant isolation strictly.
- Must use `tenantDb` wrapper only.
- Must not alter write-side behavior.
- Must not introduce background workers.
- Must keep batched reads only (no N+1).
- Output must be deterministic.
- Execution remains sequential.

### Scope Constraints
- Modify projection service and DB read helpers only.
- Route remains thin and Prisma-free.
- Route may accept optional `actorId` query param for `canComplete` calculation.
- Add integration tests for enriched projection shape and determinism.

### Required Per-Task Additions
1. `upstreamTaskIds` = runtime dependency `fromTaskId` values where `toTaskId == taskId`.
2. `downstreamTaskIds` = runtime dependency `toTaskId` values where `fromTaskId == taskId`.
3. `blockingTaskIds` = `upstreamTaskIds` whose upstream task state is not `DONE`.
4. `isReady` = `(task.state === NOT_STARTED) && (blockingTaskIds.length === 0)`.
5. `canComplete` (read-only eligibility):
   - if `actorId` omitted: `canComplete = null`
   - if `actorId` provided:
     - task state must be `NOT_STARTED`
     - actor RBAC must pass: actor has ADMIN role OR actor roleId equals `task.ownerRoleId`
     - approval policy must pass using existing 03.3 semantics in check-only mode
     - PRECONDITION gates must pass using existing minimal evaluator in check-only mode
   - do not call resolver
   - do not mutate anything
   - output boolean only (no internal reason leak)

### Required Global Additions
- `tasksTopologicalOrder`: deterministic Kahn topo order over runtime dependency graph if DAG; otherwise stable fallback order `createdAt asc, id asc`.
- `counts`:
  - `totalTasks`
  - `doneTasks`
  - `blockedTasks`
  - `readyTasks` (from `isReady`)

### Batching Requirements
- 1 query: tasks by `ecoId + tenantId`
- 1 query: dependencies by `taskIds`
- 1 query: approvals by `taskIds`
- 1 query: gates by `taskIds`
- if `actorId` provided: 1 query actor role assignments (tenant-scoped)
- no per-task queries

### Claude's Discretion
- Internal map/graph structures used for deterministic computation.
- Exact projection payload envelope arrangement as long as required fields are present and stable.

</decisions>

<specifics>
## Specific Ideas

- Reuse existing approval policy evaluator in read/check-only mode (no write or transition calls).
- Build per-task upstream/downstream adjacency maps once from batched dependency rows.
- Compute blocking/isReady/canComplete strictly from persisted rows and deterministic helper evaluation.
- Keep route input handling limited to `tenantId` and optional `actorId`.

</specifics>

<deferred>
## Deferred Ideas

- Projection-side diagnostics for why `canComplete` is false.
- Precomputed/materialized projection caches.
- Extended graph analytics beyond required counts and topo order.

</deferred>
