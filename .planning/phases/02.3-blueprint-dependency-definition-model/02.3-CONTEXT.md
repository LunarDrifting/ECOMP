# Phase 2.3: Blueprint Dependency Definition Model - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Add Prisma schema support for `TemplateDependencyDefinition` to represent dependency edges between `TemplateTaskDefinition` blueprint nodes under a `TemplateVersion`.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- Schema-only changes.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- `TemplateDependencyDefinition` must include:
  - `templateVersionId` (required)
  - `tenantId` (required)
  - `fromDefinitionId` (required)
  - `toDefinitionId` (required)
  - `type` (reuse existing `DependencyType` enum)
  - `lagMinutes` (required `Int`)
  - `createdAt`
- Must reference `TemplateVersion`.
- Must reference `TemplateTaskDefinition` for both `fromDefinitionId` and `toDefinitionId`.
- Prisma relation names must be explicit to avoid ambiguity:
  - `fromDefinition` relation uses `@relation("TemplateDependencyFrom")`
  - `toDefinition` relation uses `@relation("TemplateDependencyTo")`
- `TemplateTaskDefinition` must include required inverse relations:
  - `outgoingDependencies TemplateDependencyDefinition[] @relation("TemplateDependencyFrom")`
  - `incomingDependencies TemplateDependencyDefinition[] @relation("TemplateDependencyTo")`
- Must include `@@index([tenantId])`.
- Must include indexes on `fromDefinitionId` and `toDefinitionId`.
- No modification of existing model fields.
- Sequential execution only.
- Self-dependency (`fromDefinitionId == toDefinitionId`) is not allowed at service layer, but is not enforced in schema in this phase.

### Claude's Discretion
- Additional additive indexes only if they do not alter existing model fields.

</decisions>

<specifics>
## Specific Ideas

- Use required relation names exactly: `TemplateDependencyFrom` and `TemplateDependencyTo`.
- Add required inverse relation fields on `TemplateTaskDefinition`: `outgoingDependencies` and `incomingDependencies`.
- Keep model append-only for this phase with `createdAt` and no business logic fields.
- Ensure migration SQL reflects foreign keys to `Tenant`, `TemplateVersion`, and both `TemplateTaskDefinition` endpoints.

</specifics>

<deferred>
## Deferred Ideas

- Runtime instantiation of dependency graphs into concrete `Dependency` rows.
- Any service/API behavior for evaluating dependency constraints.
- Any refactor to existing models or enums.

</deferred>
