# Phase 4.0: Cascading Graph Resolution Engine (Deterministic, Transaction-Safe) - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Upgrade runtime dependency resolution from single-hop downstream unblocking to deterministic full cascading resolution across runtime DAG edges, while preserving all current enforcement layers and transactional safety.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must use `tenantDb` wrapper only.
- Must not introduce background workers.
- Must remain deterministic.
- Must preserve tenant isolation, RBAC, approval enforcement, gate enforcement, transition guardrails, idempotency, and transactional atomicity.
- Must not modify projection semantics (optional projection field expansion is deferred by default).
- Execution remains sequential.

### Cascading Resolver Scope
- Introduce/extend resolver as `resolveDependenciesCascade({ tenantId, taskId, tx? })`.
- Traverse runtime `Dependency` edges from `fromTaskId -> toTaskId`.
- For each downstream candidate, unblock only if:
  - all incoming upstream tasks are `DONE`
  - candidate task is currently `BLOCKED`
  - approval policy check passes (reuse existing evaluator semantics in check-only mode)
  - PRECONDITION gate checks pass (existing minimal evaluator semantics, check-only mode)
- If unblocked (`BLOCKED -> NOT_STARTED`), enqueue for further downstream cascade traversal.
- Use defensive visited set to prevent infinite traversal on corrupted data.

### Transaction and Guardrail Requirements
- `markTaskDone` remains the canonical completion path.
- `markTaskDone` keeps a single transaction boundary for `DONE` mutation + cascading resolver effects.
- Resolver accepts `tx` and must not open nested transaction.
- All unblock mutations must call transition guard `assertTransitionAllowed(BLOCKED, NOT_STARTED)`.
- No additional state mutation types allowed.
- Rollback safety tests must inject failures via Vitest mocking/spying of tx-bound service/db helper calls only (no test-only flags, env toggles, or production hooks).

### Idempotency + Deterministic Outputs
- If task already `DONE`, return deterministic no-op without cascade.
- If no downstream tasks, deterministic status `noop_no_downstream`.
- If downstream exists but none eligible, deterministic status `noop_no_eligible`.
- On work performed, status `resolved`.
- Return deterministic sorted `unblockedTaskIds`.

### Performance Constraints
- No N+1 queries.
- Batch read patterns for:
  - downstream edges for visited frontier
  - incoming dependencies for candidate tasks
  - upstream task states
  - approvals for candidate tasks
  - gates for candidate tasks
- Build adjacency and lookup maps in memory for deterministic evaluation.

### Claude's Discretion
- Exact in-memory graph traversal strategy (BFS/queue preferred for deterministic cascading).
- Exact helper function decomposition across services while preserving current architecture patterns.
- Whether `visitedTaskIds` debug output is included (only if safe and non-breaking).

</decisions>
