<task type="auto">
  <name>Task 2: Implement resolveDependenciesCascade and integrate markTaskDone transaction flow</name>
  <files>src/services/template-instantiation.service.ts, src/lib/db.ts</files>
  <action>
Replace/extend single-hop resolver with `resolveDependenciesCascade({ tenantId, taskId, tx? })`.
Behavior:
- Traverse downstream via queue/visited set.
- Evaluate candidate tasks only with runtime dependency records and batched maps.
- Candidate unblock criteria:
  - all incoming upstream tasks are DONE
  - candidate is BLOCKED
  - approval policy check passes (existing semantics, check-only mode)
  - PRECONDITION gates pass (existing minimal evaluator, check-only mode)
- For each unblock:
  - assertTransitionAllowed(BLOCKED, NOT_STARTED)
  - mutate state to NOT_STARTED
  - enqueue for further traversal
Integrate into `markTaskDone`:
- preserve pre-transaction validation order
- keep already-DONE idempotent no-op behavior (no resolver call)
- execute DONE mutation + cascade resolver in same transaction using shared tx client
- no nested transaction creation
Return deterministic structured result:
- tasksUnblocked
- unblockedTaskIds (sorted)
- status (resolved | noop_no_downstream | noop_no_eligible)
Add lightweight code comment explaining batched read strategy and N+1 avoidance.
  </action>
  <verify>
    <automated>npm run lint && npm run build</automated>
  </verify>
  <done>Cascading dependency resolution is deterministic, tx-safe, and enforcement-compliant.</done>
</task>
