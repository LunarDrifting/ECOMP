# Phase 3.1: Blueprint Graph Integrity Validation - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Validate blueprint dependency graph integrity before any runtime instantiation writes occur.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce tenant isolation strictly.
- Must use `tenantDb` wrapper only.
- Must not modify projection logic.
- Must not modify approval or gate enforcement logic.
- Must preserve idempotency.
- Must not introduce background workers.
- Sequential execution only.

### Required Guarantees
- No circular dependencies in `TemplateDependencyDefinition` blueprint graphs.
- No self-dependency edges at blueprint level.
- Deterministic instantiation failure on invalid blueprint graphs.
- No partial runtime task/dependency creation when blueprint invalid.

### Behavior Requirements
1. Add blueprint validation step inside `instantiateTemplateForEco` before:
   - ECOPlan creation
   - Task creation
   - Dependency materialization
   - State computation
2. Validation must batch-load:
   - `TemplateTaskDefinition` rows for `templateVersionId + tenantId`
   - `TemplateDependencyDefinition` rows for `templateVersionId + tenantId`
3. Validation rejects when:
   - `fromDefinitionId === toDefinitionId` (self-edge)
   - graph contains a cycle
4. Cycle detection requirements:
   - deterministic algorithm (DFS or Kahn)
   - in-memory only
   - no per-node query patterns (no N+1)
   - fail fast on detection
   - deterministic validation error mapped to HTTP 409
5. Failure behavior on invalid blueprint:
   - do not create ECOPlan
   - do not create tasks
   - do not create runtime dependencies
   - deterministic error body: `{ error: "Invalid blueprint: circular dependency detected" }`
6. Success behavior on valid blueprint:
   - continue existing instantiation flow unchanged
   - preserve idempotent no-op branch
   - preserve approval + gate enforcement logic
   - preserve dependency materialization + state computation
7. No additional projection fields.
8. No schema/model changes.

### Claude's Discretion
- Internal validation helper organization and error-code mapping strategy in service/route as long as deterministic output contract is preserved.

</decisions>

<specifics>
## Specific Ideas

- Run self-edge check and cycle check on blueprint edges immediately after batched template definitions/dependency loads.
- Use task-definition id set to build adjacency in-memory for deterministic cycle detection.
- Keep route changes minimal: map validation failures to deterministic 409 response.

</specifics>

<deferred>
## Deferred Ideas

- Rich blueprint validation diagnostics (e.g., path trace, all cycle enumeration).
- Soft-warning or lint-style blueprint diagnostics.
- Any schema/model migrations.

</deferred>
