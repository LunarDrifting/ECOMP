# Phase 3.4: Transactional Transition Atomicity - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided execution-plan specification

<domain>
## Phase Boundary

Enforce atomic, rollback-safe DONE transition behavior for task completion and dependency resolution.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must enforce strict tenant isolation.
- Must preserve idempotency behavior.
- Must keep projection logic unchanged.
- Execution remains sequential.

### Required Guarantees
- `markTaskDone` wraps DONE mutation + resolver effects in one transaction.
- Pre-mutation validations run before transaction starts.
- Any failure in transactional segment rolls back all writes.
- Resolver uses caller transaction context (no nested transaction ownership).
- `tenantDb` must accept an optional Prisma transaction client (`tx`) and bind wrapper operations to either base `prisma` or provided `tx`.
- No service-layer code may reference base `prisma` client inside transaction scope.

### Claude's Discretion
- Exact transactional wrapper shape in `tenantDb`.
- Internal function signatures for passing transaction-scoped context.

</decisions>

<specifics>
## Specific Ideas

- Add transaction-aware tenantDb constructor/wrapper with optional `tx` parameter.
- Reuse existing resolver logic with injected transactional db context.
- Keep transactional boundary narrow and deterministic.

</specifics>

<deferred>
## Deferred Ideas

- Cross-service orchestration transactions.
- Retry/backoff semantics for serialization conflicts.

</deferred>
