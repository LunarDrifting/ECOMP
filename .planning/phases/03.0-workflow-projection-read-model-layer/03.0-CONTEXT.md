# Phase 3.0: Workflow Projection / Read Model Layer - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Add a read-only, tenant-scoped workflow projection service and endpoint that derives workflow view data for a single ECO.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce tenant isolation strictly.
- Must use `tenantDb` wrapper only.
- Must not modify write-side logic.
- Must not introduce background workers.
- Sequential execution only.

### Behavior Requirements
1. Add service method `getWorkflowProjection({ tenantId, ecoId })`.
2. Load:
   - Tasks for ECO
   - Runtime Dependencies
   - Approvals
   - Gates
   - All reads must be batched:
     - tasks in one query
     - dependencies in one query
     - approvals in one query using taskIds
     - gates in one query using taskIds
   - No per-task N+1 query patterns.
3. Return structured projection:
   - `tasks` (with state, approvalPolicy, gate presence)
   - `dependencies` (`fromTaskId`, `toTaskId`)
   - derived flags:
     - `isBlocked = task.state === "BLOCKED"`
     - `requiresApproval = task.approvalPolicy !== "NONE"`
     - `requiresPrecondition = existence of at least one PRECONDITION gate row for task`
4. Do not mutate any state.
5. Projection must not recompute business logic:
   - no dependency evaluation
   - no approval evaluation
   - no gate condition evaluation
   - reflect persisted state + relationship presence only.
6. Add endpoint: `GET /api/ecos/{id}/projection`.
7. Route must remain thin and Prisma-free.

### Claude's Discretion
- Exact projection payload envelope shape beyond required fields, as long as required task/dependency/derived flag data is present.
- Whether derived `isBlocked` is computed from task state, incoming dependency presence, or both, provided it is deterministic and documented.

</decisions>

<specifics>
## Specific Ideas

- Add tenantDb read helpers for eco-scoped tasks/dependencies and task-scoped approvals/gates in batch.
- Build task projection rows using in-memory maps keyed by task id for approvals/gates/incoming-dependency checks.
- Build derived flags directly from persisted task fields and relationship presence only (no rule re-evaluation).
- Keep projection service read-only and separate from completion/instantiation write paths.

</specifics>

<deferred>
## Deferred Ideas

- Background projection materialization.
- Cached denormalized read models.
- Cross-ECO or portfolio-level projection aggregation.
- Any schema/model migrations.

</deferred>
