**Gathered:** 2026-02-27
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Upgrade `instantiateTemplateForEco` so it creates a full Task tree from `TemplateTaskDefinition` rows for the selected `TemplateVersion`.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- No modifications to existing models.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce strict tenant isolation.
- Must use `tenantDb` wrapper only (no raw Prisma).
- Must not evaluate gates or approvals in this phase.
- Must not modify existing Task model structure.
- Sequential execution only.

### Behavior Requirements
1. Load all `TemplateTaskDefinition` rows for given `templateVersionId` and `tenantId`.
2. Validate TemplateVersion tenant ownership.
3. Ensure ECOPlan exists before creating tasks.
4. Create Tasks in deterministic parent-first order.
5. Map `parentDefinitionId -> parentTaskId` correctly.
6. Return structured result including:
   - `ecoPlanId`
   - number of tasks created
   - array of created task IDs
7. Enforce parent-first creation safety:
   - No child Task is created before its parent TaskId exists.
   - Use either null-parent-first + iterative child creation or iterative passes until parents resolve.
8. Enforce idempotency:
   - If ECOPlan exists and tasks already exist for ECO, return deterministic conflict or no-op response.
   - Repeated instantiate calls must not duplicate task trees.
9. Blueprint instantiation must not mutate existing Tasks.

### Claude's Discretion
- Exact deterministic ordering strategy among siblings (e.g., createdAt then id) as long as stable.
- Error behavior for malformed definition graphs (e.g., missing parent reference).

</decisions>
