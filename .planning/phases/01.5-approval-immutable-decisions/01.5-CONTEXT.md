# Phase 1.5: Approval Immutable Decisions - Context

**Gathered:** 2026-02-27
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Add Prisma schema support for an `Approval` domain model with immutable decision semantics. This phase is schema and migration work only.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Schema-only changes; no business logic.
- No service or route modifications.
- `Approval` model must include:
  - `taskId` (required)
  - `actorId` (required; references `User.id`)
  - `tenantId` (required; references `Tenant.id`)
  - `decision` enum (`APPROVED`, `REJECTED`)
  - `comment` (nullable)
  - `createdAt`
- `Approval` must reference `Task`, `User`, and `Tenant`.
- `Approval` must include `@@index([tenantId])`.
- Decision semantics must be immutable (append-only decision records; no update semantics in schema modeling).
- Add necessary inverse relations on `Task` and `User`.
- No modification of existing fields beyond required inverse relation additions.
- Sequential execution only.

### Claude's Discretion
- Additive indexes/constraints supporting lookup patterns and append-only history semantics.
- Whether to include an `id` primary key vs composite primary key, as long as immutability semantics are preserved.

</decisions>

<specifics>
## Specific Ideas

- Use `createdAt` for decision chronology.
- Keep `Approval` append-only by not including `updatedAt`.
- Ensure relations are explicit and compatible with existing tenant/RBAC schema.
- Keep tenant-scoped query support explicit through `tenantId` and index.

</specifics>

<deferred>
## Deferred Ideas

- Any route/service/business-logic enforcement.
- Any mutation rules in application code (this phase is schema-only).
- Refactors outside required inverse relation additions.

</deferred>
