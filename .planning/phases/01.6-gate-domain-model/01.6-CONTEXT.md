# Phase 1.6: Gate Domain Model - Context

**Gathered:** 2026-02-27
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Add Prisma schema support for a `Gate` domain model aligned to the Design Spec. This phase is schema and migration work only.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- Schema-only changes.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- `Gate` model must include:
  - `taskId` (required)
  - `tenantId` (required)
  - `type` enum (`PRECONDITION`, `POSTCONDITION`)
  - `condition` (`Json`, required)
  - `createdAt`
- `Gate` must reference `Task` and `Tenant`.
- `Gate` must include `@@index([tenantId])`.
- Add required inverse relation on `Task`.
- No modification of existing fields beyond required inverse relation additions.
- Sequential execution only.

### Claude's Discretion
- Additive indexes/constraints beyond `@@index([tenantId])` if needed for safe relation performance.
- Identifier strategy (`id` key shape) as long as required fields and semantics hold.

</decisions>

<specifics>
## Specific Ideas

- Keep gate condition payload as required JSON for spec flexibility.
- Keep model append-only style by using `createdAt` and no update-oriented field unless strictly required (not required here).
- Maintain tenant-scoped data access path in schema through explicit `tenantId` relation.

</specifics>

<deferred>
## Deferred Ideas

- Any route/service/business-logic work.
- Any execution engine/runtime behavior for evaluating gates.
- Refactors outside required inverse relation additions.

</deferred>
