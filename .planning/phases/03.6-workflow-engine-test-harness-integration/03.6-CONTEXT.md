# Phase 3.6: Workflow Engine Test Harness (Integration) - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Add a deterministic automated integration test harness validating core workflow engine behavior end-to-end without altering production runtime behavior.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- No production behavior changes.
- Tests only plus minimal test helpers.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must preserve tenant isolation.
- Must not introduce background workers.
- Must keep tests deterministic (no Date.now assertions).
- Execution remains sequential.

### Framework Selection
- Choose one framework only.
- Selected framework: **Vitest** (preferred, Node-level tests).

### Required Coverage
1. Instantiation creates tasks/dependencies and `blocked + ready = tasksCreated`.
2. Completing BLOCKED task returns 409.
3. Completing NOT_STARTED task succeeds and unblocks downstream when applicable.
4. Re-completing DONE task is idempotent (`taskMarkedDone: false`).
5. Approval policy failure returns canonical 409 and no mutation.
6. Gate failure returns 409 and no mutation.
7. Cyclic blueprint is rejected pre-write with no ECOPlan/tasks/dependencies created.

### Environment Requirements
- Tests run against a test database (separate DB name via env).
- Setup/teardown should use direct Prisma in test utilities (preferred over dev seed route).
- Vitest runs must require `DATABASE_URL_TEST` (or equivalent explicit test DB env) and fail fast if missing.
- Tests must never fall back to or use `DATABASE_URL`.

### Claude's Discretion
- Exact test folder layout and helper module structure.
- Whether tests call service-layer functions directly or route handlers through API server harness, as long as behavior requirements remain end-to-end for workflow engine behavior and deterministic.

</decisions>

<specifics>
## Specific Ideas

- Add Vitest config and test script in `package.json`.
- Add deterministic fixture helpers for tenant/user/role/template/eco setup and teardown.
- Add integration tests grouped by behavior scenario with isolated DB data per test.
- Add explicit test bootstrap guard that aborts immediately when `DATABASE_URL_TEST` is absent.
- For cyclic blueprint case, verify DB contains no partial writes by asserting no ECOPlan/Task/Dependency persistence.

</specifics>

<deferred>
## Deferred Ideas

- Parallelized test execution.
- Browser/UI test coverage.
- Snapshot-based projection regression suites.

</deferred>
