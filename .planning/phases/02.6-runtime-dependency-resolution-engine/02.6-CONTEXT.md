# Phase 2.6: Runtime Dependency Resolution Engine - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Implement runtime dependency resolution so downstream blocked tasks can be unblocked when an upstream task reaches `DONE`.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce tenant isolation strictly.
- Must use `tenantDb` wrapper only (no raw Prisma in service/route).
- Must not evaluate gates or approvals.
- Must not introduce background workers or async orchestration.
- Must preserve idempotency guarantees.
- Sequential execution only.

### Behavior Requirements
1. Add service method `resolveDependenciesForTask({ tenantId, taskId })`.
   - Implement as a standalone service method.
   - Must be manually invocable in this phase.
   - Do not auto-integrate into instantiation flow.
   - Do not introduce a new transition engine yet.
2. When called after a task is marked `DONE`:
   - Load downstream tasks via runtime `Dependency` rows.
   - For each downstream task:
     - Check all incoming runtime dependencies.
     - If all upstream tasks are `DONE` -> set state to `NOT_STARTED` (only if currently `BLOCKED`).
   - Upstream dependency checks must load required upstream `Task.state` values in batch.
   - Avoid N+1 query patterns.
   - Use `tenantDb` wrapper only.
3. Do not mutate tasks that are not `BLOCKED`.
4. Do not re-evaluate entire graph; only affected downstream tasks.
5. Return structured result:
   - `tasksUnblocked` count
   - `unblockedTaskIds`
6. Resolver must not be triggered automatically during instantiation.

### Claude's Discretion
- Exact error semantics when task is missing or not tenant-scoped.
- Whether resolver executes in a single pass over immediate downstream tasks (required minimum scope) with deterministic ordering.
- Placement of resolver method in existing service files for this codebase (no dedicated task-state service exists yet).

</decisions>

<specifics>
## Specific Ideas

- Add tenantDb helpers for downstream dependency lookup and batched incoming upstream-state checks.
- Keep resolver narrow: only direct downstream tasks of completed task are considered in this phase.
- Preserve idempotency by only mutating tasks currently `BLOCKED`; repeated resolver calls should become no-ops after unblocking.
- Expose the resolver as a standalone service-layer method callable manually for now.

</specifics>

<deferred>
## Deferred Ideas

- Recursive/multi-hop graph propagation.
- Event-driven or background dependency resolution.
- Gate/approval aware readiness logic.
- Any schema/model migrations.

</deferred>
