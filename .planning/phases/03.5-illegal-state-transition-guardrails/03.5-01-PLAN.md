Introduce deterministic illegal state transition guardrails to prevent invalid task state mutations.

Purpose:
Ensure state transitions follow a defined matrix and prevent corruption from accidental or future code paths.

Output:
Centralized transition validation layer with deterministic 409 rejection behavior.
</objective>

<task type="auto">
  <name>Task 1: Define canonical state transition matrix</name>
  <files>src/services/state-transition.service.ts</files>
  <action>
Create a new service file that defines allowed Task.state transitions:

Allowed transitions:

NOT_STARTED → DONE
BLOCKED → NOT_STARTED
NOT_STARTED → BLOCKED (instantiation only)
BLOCKED → DONE (explicitly disallowed)
DONE → any (disallowed)
DONE → DONE (idempotent no-op allowed but not mutation)

Export:

- canTransition(fromState, toState)
- assertTransitionAllowed(fromState, toState)

Reject illegal transitions with canonical message:
"Illegal state transition"

Do not introduce side effects.
  </action>
  <verify>
    <automated>npm run lint</automated>
  </verify>
  <done>Transition matrix exists and can validate transitions deterministically.</done>
</task>

<task type="auto">
  <name>Task 2: Guard markTaskDone and resolver state updates</name>
  <files>src/services/template-instantiation.service.ts</files>
  <action>
Before any state mutation:

- Call assertTransitionAllowed(currentState, targetState)

Enforce:
- DONE transition must only originate from NOT_STARTED
- BLOCKED tasks cannot transition directly to DONE
- DONE → DONE remains idempotent no-op (no mutation)

Resolver unblocking:
- Validate BLOCKED → NOT_STARTED transition before mutation

Do not change validation ordering (BLOCKED, approval, gate checks remain pre-transaction).
Do not introduce nested transactions.
  </action>
  <verify>
    <automated>npm run lint && npm run build</automated>
  </verify>
  <done>All state mutations pass through transition guard.</done>
</task>
