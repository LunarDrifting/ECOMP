## Phase Boundary

Introduce deterministic service-layer guardrails preventing illegal `Task.state` mutations through a canonical transition matrix.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must enforce tenant isolation.
- Must preserve existing transition orchestration semantics.
- Execution remains sequential.

### Required Guarantees
- Canonical transition matrix defines valid `Task.state` transitions.
- State mutations are validated before mutation.
- Illegal transitions reject deterministically with 409.
- DONE transition remains only via `markTaskDone`.
- Resolver path uses transition guardrails.

### Claude's Discretion
- Internal shape of transition matrix constants/helpers.
- How db helper restrictions are encoded while preserving existing service patterns.

</decisions>

<specifics>
## Specific Ideas

- Add dedicated `state-transition.service.ts` with pure transition checks.
- Gate all service-level state updates through `assertTransitionAllowed`.
- Keep `DONE -> DONE` as idempotent no-op behavior (no mutation).

</specifics>
