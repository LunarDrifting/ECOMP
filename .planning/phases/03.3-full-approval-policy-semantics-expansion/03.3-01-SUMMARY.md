# Phase 3.3 Plan 01 Summary

**Expanded `markTaskDone` to deterministic policy-specific approval semantics for NONE/SINGLE/SEQUENTIAL/PARALLEL/QUORUM with canonical 409 failure behavior.**

## Accomplishments

- Extended `tenantDb` policy-evaluation reads:
  - `approval.listByTaskId` now includes `actorId` + `decision` for one-shot policy evaluation data.
  - Added `userRole.listRoleAssignmentsByUserIds(userIds)` for single batched actor-role resolution.
- Implemented deterministic approval policy evaluator in `markTaskDone`:
  - `NONE`: pass without approval requirement.
  - `SINGLE`: requires >=1 `APPROVED`.
  - `SEQUENTIAL`: rejects if any `REJECTED`; otherwise requires >=1 `APPROVED`.
  - `PARALLEL` (minimal): requires >=1 `APPROVED` from actor with `roleId == task.ownerRoleId`.
  - `QUORUM`: requires >=2 `APPROVED` from distinct `actorId` values.
- Added canonical failure handling:
  - Throws exact message `Approval policy requirements not satisfied` on policy failure.
  - Keeps no mutation and no resolver invocation on policy failure.
- Updated completion route mapping:
  - canonical policy failure now maps deterministically to HTTP 409.
