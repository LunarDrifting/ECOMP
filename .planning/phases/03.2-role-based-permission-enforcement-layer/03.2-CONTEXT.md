# Phase 3.2: Role-Based Permission Enforcement Layer - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Introduce deterministic, tenant-scoped RBAC checks for task completion and approval creation in the service layer.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce tenant isolation strictly.
- Must use `tenantDb` wrapper only.
- Must not modify projection layer.
- Must not introduce background workers.
- Must preserve idempotency.
- Sequential execution only.

### Required Scope
1. `markTaskDone` completion authorization:
   - accept `actorId`
   - allow only task owner role or `ADMIN` role
   - reject unauthorized with deterministic 403
2. Approval creation authorization:
   - allow only users whose role matches `task.ownerRoleId`
   - reject unauthorized with deterministic 403
3. Route handling:
   - update `POST /api/tasks/{id}/complete` to require `actorId`
   - keep route thin and Prisma-free
   - map authorization failures to 403
4. Preserve behavior:
   - do not change approval or gate semantics
   - do not alter instantiation flow
   - do not add fine-grained permission matrix in this phase

### Required Guarantees
- Authorization checks occur before writes.
- Already-DONE no-op behavior remains preserved.
- No state mutation on authorization failure.
- Deterministic 403 mapping for authorization failures.

### Claude's Discretion
- Exact helper organization in `tenantDb` and services.
- Whether approval creation authorization is exposed through a dedicated route in this phase, as long as route remains thin and service-enforced.

</decisions>

<specifics>
## Specific Ideas

- Add tenant-scoped user-role lookup helpers to avoid raw Prisma usage.
- Add service-level actor-role authorization helper reused by completion and approval creation paths.
- Keep deterministic error strings for route-level 403 mapping.

</specifics>

<deferred>
## Deferred Ideas

- Fine-grained role-permission matrix.
- Role hierarchy and delegated permissions.
- Policy-specific approval semantics beyond current phase scope.

</deferred>
