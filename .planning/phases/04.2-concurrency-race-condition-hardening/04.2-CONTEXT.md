# Phase 4.2: Concurrency & Race Condition Hardening (Deterministic, Transaction-Safe) - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Validate and harden workflow completion behavior under concurrent requests and interleaving transactions, while preserving existing transactional guarantees and enforcement semantics.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- No production behavior changes beyond safe concurrency hardening.
- Must preserve tenant isolation, RBAC, approval semantics, gate enforcement, and state-transition guardrails.
- Must preserve atomic transaction boundary in `markTaskDone`.
- Must not introduce background workers.
- Tests must be deterministic (no timing-based assertions).
- Execution remains sequential.

### Strategy Constraint
- Prefer minimal code changes.
- First prove correctness with current atomicity using concurrent integration tests.
- Add hardening only if tests expose race behavior.

### Required Concurrent Behaviors
1. Concurrent completion of same `NOT_STARTED` task:
   - exactly one marks `DONE`
   - second returns deterministic no-op/idempotent/conflict outcome
   - no partial state and no duplicate downstream unblocks
2. Concurrent completion of two upstream prerequisites for same downstream:
   - downstream becomes `NOT_STARTED` exactly once when both upstream are `DONE`
   - no duplicate unblocks
   - deterministic final state
3. Concurrent completion attempts on `BLOCKED` task:
   - all remain deterministic rejection

### Hardening Options (only if required)
- Atomic conditional update semantics (e.g. update where `state='NOT_STARTED'`).
- Row-level locking approach only if required and only within existing transaction discipline.
- Cascade resolver must stay within same tx and evaluate tx-visible state.

### Test Constraints
- Use Vitest integration tests with concurrent invocation (`Promise.all`) via route or service.
- Assert DB end-state and deterministic response classification.
- No sleeps/timeouts or timing-dependent assertions.

### Claude's Discretion
- Exact response classification expectations for the losing concurrent request (`noop_already_done` vs deterministic conflict), as long as deterministic and validated.
- Whether to run concurrency tests through route handlers or service method directly, provided coverage is end-to-end for completion semantics.

</decisions>

<specifics>
## Specific Ideas

- Introduce helper assertion utilities in integration tests to count response status classes and validate exactly-one effect.
- Validate downstream task state and returned `unblockedTaskIds` for duplicate prevention.
- If hardening is needed, add minimal tx-safe helper in `tenantDb` rather than scattering conditional logic.

</specifics>

<deferred>
## Deferred Ideas

- High-volume load testing.
- Multi-process chaos injection harness.
- Cross-tenant concurrency stress suites.

</deferred>
