# Phase 1.3: Task + Dependency Domain Models - Context

**Gathered:** 2026-02-27
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Add Prisma schema support for Task and Dependency domain models only. This phase is data-modeling and migration work; no API/service/business-logic changes.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- Use existing stack only: Next.js App Router + Prisma v6 + PostgreSQL.
- Follow `docs/ARCHITECTURE_CONTRACT.md`.
- Schema-only changes; no business logic.
- No refactor or modification of existing model definitions.
- All models tenant-scoped where applicable.
- Use Prisma migration discipline (schema change + migration files, no direct DB edits).
- Sequential execution only.
- Do not change any existing enum values.

### Required Task Model Shape
- `tenantId` (required; relation to `Tenant.id`)
- `ecoId`
- `parentTaskId` (nullable)
- `taskLevel` enum (`TaskLevel`)
- `name`
- `ownerRoleId`
- `state` enum (`TaskState`)
- `visibility` enum (`TaskVisibility`)
- `approvalPolicy` enum (`ApprovalPolicy`)
- `clockMode` enum (`ClockMode`)
- `createdAt`, `updatedAt`

### Required Dependency Model Shape
- `fromTaskId`
- `toTaskId`
- `type` enum (`DependencyType`)
- `lagMinutes`

### Required Enum Definitions (Explicit)
- `TaskLevel`: `MILESTONE`, `STEP`, `SUBSTEP`
- `TaskState`: `NOT_STARTED`, `IN_PROGRESS`, `BLOCKED`, `PENDING_APPROVAL`, `DONE`, `NOT_REQUIRED`, `CANCELLED`
- `TaskVisibility`: `INTERNAL_ONLY`, `CUSTOMER_VISIBLE`, `CUSTOMER_ACTIONABLE`
- `ApprovalPolicy`: `NONE`, `SINGLE`, `SEQUENTIAL`, `PARALLEL`, `QUORUM`
- `ClockMode`: `ACTIVE`, `WAITING_ON_CUSTOMER`, `WAITING_ON_SUPPLIER`, `WAITING_INTERNAL`
- `DependencyType`: `FINISH_TO_START`, `START_TO_START`

### Claude's Discretion
- Index placement and relation naming details, without changing existing models or existing enum values.

</decisions>

<specifics>
## Specific Ideas

- Keep ownership and relations explicit in Prisma relations (`Task` self-reference, `Task` to `Role`, `Task` to `ECO`, `Dependency` edges to `Task`).
- Ensure compatibility with current multi-tenant/RBAC foundation.
- `Task.tenantId` must reference `Tenant.id` directly.

</specifics>

<deferred>
## Deferred Ideas

- Any route handlers, service-layer logic, authorization logic changes, or workflow runtime behavior.
- State transition enforcement in code.
- Query optimizations beyond basic relational/index setup.

</deferred>
