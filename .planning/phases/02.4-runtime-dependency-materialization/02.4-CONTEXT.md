# Phase 2.4: Runtime Dependency Materialization - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Upgrade runtime instantiation so it materializes concrete `Dependency` rows from `TemplateDependencyDefinition` during ECO template instantiation.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce tenant isolation strictly.
- Must use `tenantDb` wrapper only (no raw Prisma in service/route).
- Must not evaluate gates or approvals.
- Must not mutate existing tasks.
- Must preserve idempotency behavior.
- Sequential execution only.

### Behavior Requirements
1. Load `TemplateDependencyDefinition` rows for `templateVersionId + tenantId`.
2. Map `fromDefinitionId` and `toDefinitionId` to created `taskId`s using existing `definitionId -> taskId` map.
3. Create runtime `Dependency` rows with:
   - `tenantId`
   - `fromTaskId`
   - `toTaskId`
   - `type`
   - `lagMinutes`
4. Reject self-dependency at service layer (`fromTaskId == toTaskId`).
5. If ECOPlan exists and tasks already exist, do not create duplicate dependencies.
6. Return structured result including:
   - `dependenciesCreated`
   - `createdDependencyIds`
7. Prevent duplicate runtime dependency creation within the same instantiation:
   - either deduplicate blueprint edges before creation, or
   - check existing dependency rows for `(fromTaskId, toTaskId)` before inserting.

### Claude's Discretion
- Exact deterministic error semantics for unresolved dependency definitions (e.g., missing definition mapping).
- Whether endpoint status text changes as long as route remains thin and service-driven.

</decisions>

<specifics>
## Specific Ideas

- Extend tenantDb with tenant-scoped `TemplateDependencyDefinition` list and `Dependency` create/list helpers.
- Reuse existing task-creation map in instantiation service to materialize dependencies after task creation completes.
- Preserve idempotent no-op branch by returning zero dependency creation when tasks already exist.
- Add deterministic duplicate-edge safeguard so repeated/duplicate blueprint edges cannot produce duplicate runtime `Dependency` rows.

</specifics>

<deferred>
## Deferred Ideas

- Gate evaluation logic.
- Approval evaluation logic.
- Any schema/model migrations.
- Mutation or reconciliation of existing task/dependency graphs.

</deferred>
