# Phase 2.8: Approval Enforcement Before DONE - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning
**Source:** User-provided phase goal and constraints

<domain>
## Phase Boundary

Enforce approval requirements inside `markTaskDone` before allowing a task to transition to `DONE`.

</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- No schema changes.
- Service-layer only.
- Must follow `docs/ARCHITECTURE_CONTRACT.md`.
- Must enforce tenant isolation strictly.
- Must use `tenantDb` wrapper only (no raw Prisma in route).
- Must not evaluate gates yet.
- Must preserve idempotency.
- Must not introduce background workers.
- Sequential execution only.

### Behavior Requirements
1. In `markTaskDone`:
   - If `approvalPolicy === NONE` -> allow transition.
   - Otherwise:
     - Load `Approval` rows for task (tenant-scoped).
     - Require at least one Approval with `decision === APPROVED`.
     - If none -> reject with deterministic `409`.
   - For this phase, all non-NONE policies (`SINGLE`, `SEQUENTIAL`, `PARALLEL`, `QUORUM`) are treated identically with the same minimum check above.
   - Do not enforce ordering, quorum thresholds, or role-based approval constraints in this phase.
   - Do not implement rejection semantics beyond simple block when no APPROVED row exists.
2. If rejected:
   - Do not update state.
   - Do not call resolver.
3. If allowed:
   - Proceed with DONE update.
   - Call `resolveDependenciesForTask`.
4. Preserve existing:
   - `BLOCKED` rejection.
   - Already-DONE idempotency.
5. Return structured result including:
   - `taskMarkedDone`
   - `tasksUnblocked`
   - `unblockedTaskIds`
   - `approvalCheckPassed` (boolean)

### Claude's Discretion
- Exact error/status text for approval rejection as long as it is deterministic and mapped to 409.
- Whether approval check runs before or after BLOCKED/already-DONE checks, provided behavior remains consistent and idempotent guarantees are preserved.

</decisions>

<specifics>
## Specific Ideas

- Add tenantDb helper to list approvals by task id with tenant scoping.
- Keep route thin; service decides completion eligibility and returns approval check signal.
- Return `approvalCheckPassed=false` for deterministic rejection/no-op paths as appropriate.
- Explicitly defer full approval-policy semantics (`SINGLE`/`SEQUENTIAL`/`PARALLEL`/`QUORUM`) to a later phase.

</specifics>

<deferred>
## Deferred Ideas

- Gate enforcement before DONE.
- Rich approval-policy semantics beyond existence of one APPROVED row (ordering, quorum thresholds, role constraints).
- Background reconciliation/worker-based approval processing.
- Any schema/model migrations.

</deferred>
